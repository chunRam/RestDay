# RestDay AI Agent View Guide

이 문서는 다른 AI 에이전트가 RestDay 앱의 기능, 사용 흐름, 첫 화면 View 구조를 이해하고 구현할 수 있도록 만든 화면 설계 기준이다.

## 1. 핵심 판단

RestDay를 실행했을 때 사용자가 처음 만나야 하는 화면은 기능 목록 화면이 아니다.

첫 화면은 다가오는 휴일을 기준으로 지금 해야 할 행동을 알려주는 `HomeView`여야 한다.

핵심 질문:

> 다가오는 휴일이 있는데, 지금 준비할까요?

RestDay는 휴일 관리 앱이 아니라 휴일 전 의사결정 앱이다. 따라서 시작점은 항상 활동 카테고리가 아니라 다가오는 휴일이어야 한다.

## 2. 앱 기능 구조

RestDay의 기능은 크게 5개로 본다.

1. 휴일 확인
   - 사용자가 준비할 휴일을 정한다.
   - 초기 버전에서는 자동 인식보다 수동 날짜 등록을 우선한다.

2. 휴일 전 질문
   - 사용자의 현재 상태, 에너지, 원하는 방향을 짧게 묻는다.
   - RestDay의 핵심 기능이다.

3. 하루 방향 결정
   - 사용자가 이번 휴일을 어떻게 보낼지 결정한다.
   - 예시는 쉬는 날, 정리하는 날, 회복하는 날, 집중하는 날 등이 될 수 있다.
   - 단, 이 분류는 고정 enum처럼 박아두지 않는다.

4. 계획 생성
   - 선택 결과를 바탕으로 실행 가능한 작은 일정표를 만든다.
   - 초기 구현은 규칙 기반 추천으로 충분하다.

5. 실행 체크와 회고
   - 휴일 당일에는 계획을 체크한다.
   - 휴일 종료 후 만족도와 한 줄 회고를 남긴다.

## 3. 앱 사용 흐름

사용자가 앱을 켰을 때의 기본 흐름은 다음과 같다.

1. 앱 실행
2. `HomeView`에서 다가오는 휴일 확인
3. 휴일이 없으면 `HolidayRegisterView`로 이동
4. 휴일이 있으면 `HolidayDecisionView`에서 질문에 답변
5. 답변을 바탕으로 `PlanPreviewView`에서 하루 계획 확인
6. 휴일 당일에는 `ExecutionView`에서 체크
7. 휴일 종료 후 `ReviewView`에서 만족도와 회고 저장
8. 다시 `HomeView`로 돌아와 다음 휴일 준비

## 4. 필수 View 구성

### 4.1 HomeView

앱 실행 시 첫 화면이다.

역할:

- 다음 휴일을 보여준다.
- 휴일까지 남은 날짜를 보여준다.
- 현재 사용자가 해야 할 다음 행동 하나를 보여준다.
- 가장 중요한 CTA 버튼 하나를 제공한다.

상태별 CTA:

| 상태 | CTA |
|---|---|
| 등록된 휴일 없음 | 휴일 등록하기 |
| 휴일은 있음, 계획 없음 | 이번 휴일 준비하기 |
| 계획 있음, 휴일 전 | 계획 확인하기 |
| 휴일 당일 | 오늘 계획 체크하기 |
| 휴일 지남, 회고 없음 | 휴일 회고하기 |

`HomeView`는 RestDay의 관제 화면이다. 여러 기능을 한꺼번에 나열하지 말고 사용자가 지금 해야 할 다음 행동 하나를 강조한다.

### 4.2 HolidayRegisterView

휴일 날짜를 입력하는 화면이다.

초기 MVP에서는 캘린더 연동보다 날짜 직접 선택을 우선한다.

필수 요소:

- 휴일 날짜 선택
- 휴일 이름 또는 메모 입력
- 저장 버튼

### 4.3 HolidayDecisionView

휴일 전 질문 화면이다.

이 화면이 RestDay의 핵심이다. 사용자가 "이번 휴일을 어떻게 보낼지" 결정하게 만들어야 한다.

질문 예시:

- 오늘은 몸을 쉬게 하고 싶은가, 밀린 일을 처리하고 싶은가?
- 혼자 보내고 싶은가, 누군가와 함께 보내고 싶은가?
- 에너지가 높은 편인가, 낮은 편인가?
- 오늘 꼭 끝내고 싶은 일이 있는가?

주의:

- 처음부터 활동 카테고리만 나열하지 않는다.
- 사용자의 상태와 의도를 먼저 묻는다.
- 선택지는 나중에 바뀔 수 있는 데이터 구조로 둔다.

### 4.4 PlanPreviewView

생성된 하루 계획을 보여주는 화면이다.

필수 요소:

- 시간대별 계획
- 핵심 할 일
- 계획 확정 버튼
- 필요하면 가벼운 수정 기능

초기 버전에서는 복잡한 AI 계획보다 간단한 규칙 기반 계획이 적합하다.

### 4.5 ExecutionView

휴일 당일 체크 화면이다.

필수 요소:

- 계획 항목 목록
- 완료 체크
- 진행률 또는 완료 개수

복잡한 타이머, 통계, 알림보다 체크리스트가 먼저다.

### 4.6 ReviewView

휴일 종료 후 회고 화면이다.

필수 요소:

- 만족도
- 잘한 점
- 다음 휴일에 바꿀 점
- 저장 버튼

회고는 길게 쓰게 만들 필요가 없다. 만족도와 한 줄 메모만으로도 MVP에 충분하다.

### 4.7 HistoryView

이전 휴일 기록을 보는 화면이다.

MVP 필수는 아니다. 핵심 흐름이 완성된 뒤 추가한다.

## 5. 구현 우선순위

초기 구현 우선순위는 다음과 같다.

1. `HomeView`
2. `HolidayRegisterView`
3. `HolidayDecisionView`
4. `PlanPreviewView`
5. `ExecutionView`
6. `ReviewView`
7. `HistoryView`

`HistoryView`는 후순위다. 먼저 사용자가 휴일을 등록하고, 결정하고, 계획을 확인하고, 체크하고, 회고하는 흐름이 끝까지 작동해야 한다.

## 6. 구현 시 피해야 할 방향

- 첫 화면을 활동 카테고리 선택 화면으로 만들지 않는다.
- 처음부터 기능 목록, 통계, 기록 리스트를 전면에 두지 않는다.
- `휴식`, `운동`, `독서`, `자기계발` 같은 분류를 고정된 앱 구조로 박아두지 않는다.
- 캘린더 연동, Firebase, AI API를 MVP의 필수 조건으로 두지 않는다.
- 화면을 많이 만드는 것보다 하나의 의사결정 흐름이 끝까지 작동하는 것을 우선한다.

## 7. 최종 기준

좋은 RestDay 화면 구조는 사용자가 앱을 켰을 때 아래 질문에 바로 답하게 만든다.

> 내 다음 휴일은 언제이고, 지금 나는 무엇을 준비해야 하는가?

이 질문에 답하지 못하는 화면은 첫 화면으로 적합하지 않다.
