# RestDay 제품 방향 문서

## 1. 앱 정의

### 앱 이름

RestDay

### 한 줄 설명

RestDay는 휴일을 아무 생각 없이 흘려보내지 않도록, 사용자가 휴일 전에 자신의 상태와 의도를 정하고 실행 가능한 하루 계획으로 옮기게 돕는 휴일 의사결정 앱이다.

### 가장 근본이 되는 기능

사용자가 다가오는 휴일을 앞두고 "이번 휴일을 어떻게 보낼지" 결정하도록 만드는 것이다.

RestDay의 핵심은 많은 기능을 제공하는 것이 아니라, 휴일 전에 아래 흐름을 자연스럽게 만들도록 돕는 데 있다.

1. 다가오는 휴일을 인식한다.
2. 사용자의 현재 상태와 휴일 의도를 묻는다.
3. 사용자가 하나의 방향을 선택한다.
4. 선택한 방향을 실행 가능한 하루 계획으로 바꾼다.
5. 휴일이 끝난 뒤 만족도와 다음 개선점을 남긴다.

이 흐름이 유지된다면 세부 기능, 분류, 화면 구조는 구현 과정에서 변경할 수 있다.

---

## 2. 문제 정의

### 현재 문제

현대인에게 휴일은 회복과 재정비를 위한 중요한 시간이다. 하지만 실제로는 휴일이 되어도 무엇을 해야 할지 정하지 못해 늦잠, 휴대폰 사용, 무의미한 소비로 하루를 보내고, 하루가 끝난 뒤 허무함과 후회를 느끼는 경우가 많다.

이 문제가 반복되면 자기효능감 저하, 생활 리듬 붕괴, 업무 및 학업 집중력 감소, 장기적인 무기력감으로 이어질 수 있다.

### 해결해야 할 지점

RestDay가 해결해야 할 핵심 문제는 "휴일을 잘 보내는 방법을 알려주는 것"이 아니라 "휴일이 오기 전에 사용자가 자신의 휴일 방향을 결정하게 만드는 것"이다.

따라서 구현은 기능 목록을 많이 만드는 방향보다, 사용자가 미리 생각하고 선택하고 실행하게 만드는 흐름을 우선해야 한다.

## 3. 제품 원칙

### 고정해야 하는 것

- RestDay는 휴일 당일에 급하게 고민하는 앱이 아니라, 휴일 전에 미리 준비하게 만드는 앱이다.
- 앱의 중심 경험은 계획표 작성이 아니라 휴일 방향 결정이다.
- 사용자는 복잡한 설정 없이 짧은 질문에 답하고 바로 실행 가능한 계획을 받아야 한다.
- 계획은 사용자가 실제로 실행할 수 있을 만큼 작고 구체적이어야 한다.
- 휴일이 끝난 뒤에는 결과를 평가하고 다음 휴일에 반영할 수 있어야 한다.

### 고정하지 않아야 하는 것

- 휴일 유형의 개수와 이름
- 활동 카테고리의 종류
- 화면 개수와 화면 이름
- 추천 로직의 세부 방식
- 캘린더 연동 여부
- AI API 사용 여부
- 기록 항목의 세부 구성

위 항목은 초기 구현에서 확정하지 않는다. 실제 UI와 사용자 흐름을 만들면서 더 자연스러운 구조가 보이면 수정할 수 있다.

---

## 4. 핵심 사용자 흐름

RestDay의 기본 흐름은 다음과 같다.

1. 사용자가 다가오는 휴일 또는 비어 있는 날을 확인한다.
2. 앱이 휴일 전에 짧은 질문을 제시한다.
3. 사용자가 현재 상태와 원하는 휴일 방향을 선택한다.
4. 앱이 선택 내용을 바탕으로 작은 실행 계획을 제안한다.
5. 사용자가 휴일 동안 계획을 체크한다.
6. 휴일 후 만족도와 개선점을 기록한다.

이 흐름은 구현의 기준이지만, 반드시 여섯 개 화면으로 나누어야 한다는 뜻은 아니다. 한 화면 안에서 단계적으로 진행해도 되고, 필요한 경우 더 작게 나누어도 된다.

---

## 5. 기능 후보

아래 기능들은 확정 목록이 아니라 구현 후보이다. 바이브 코딩 과정에서 더 단순한 구조가 적합하면 줄이고, 더 필요한 흐름이 보이면 추가한다.

| 후보 | 목적 | 구현 판단 기준 |
|---|---|---|
| 휴일 입력 | 사용자가 준비할 날짜를 정한다 | 자동 인식보다 수동 입력이 빠르면 수동 입력부터 구현한다 |
| 휴일 자동 인식 | 비어 있는 날이나 여유 있는 날을 찾는다 | 일정 데이터가 충분하지 않으면 MVP에서는 제외할 수 있다 |
| 사전 질문 | 휴일 전에 의도를 정하게 한다 | 가장 먼저 구현해야 할 핵심 기능이다 |
| 휴일 방향 추천 | 사용자의 상태에 맞는 방향을 제안한다 | 고정 카테고리보다 질문 응답 기반으로 유연하게 구성한다 |
| 하루 계획 생성 | 선택한 방향을 시간대별 행동으로 바꾼다 | 처음에는 규칙 기반 추천으로 충분하다 |
| 실행 체크 | 계획을 실제 행동으로 연결한다 | 체크리스트, 진행률, 완료 표시 중 가장 단순한 방식부터 구현한다 |
| 회고 기록 | 다음 휴일에 개선할 단서를 남긴다 | 만족도와 한 줄 메모만으로도 시작할 수 있다 |
| 동기부여 메시지 | 실행을 돕는다 | 핵심 흐름을 방해하지 않을 때만 추가한다 |

## 6. 분류 설계 원칙

휴일 유형과 활동 분류는 초기에 고정하지 않는다.

예를 들어 "휴식", "생산적인 날", "자기계발", "정리하는 날" 같은 분류는 참고용일 뿐이다. 구현 중에는 사용자가 실제로 선택하기 쉬운 표현을 우선한다.

분류를 만들 때는 아래 기준을 따른다.

- 사용자가 바로 이해할 수 있는 말인가?
- 선택지가 너무 많아 결정이 어려워지지 않는가?
- 선택 결과가 실제 계획 생성에 영향을 주는가?
- 나중에 새로운 선택지를 추가해도 구조가 무너지지 않는가?

초기 버전에서는 고정 카테고리보다 질문 기반 구조를 우선한다.

예시 질문:

- 오늘은 몸을 쉬게 하고 싶은가, 밀린 일을 처리하고 싶은가?
- 혼자 보내고 싶은가, 누군가와 함께 보내고 싶은가?
- 에너지가 높은 편인가, 낮은 편인가?
- 오늘 꼭 끝내고 싶은 일이 있는가?

이 질문들의 답을 조합해 휴일 방향과 계획을 만들 수 있다.

## 7. MVP 구현 범위

초기 프로토타입은 아래 범위만으로도 충분하다.

1. 사용자가 휴일 날짜를 직접 등록한다.
2. 앱이 휴일 전 질문을 제공한다.
3. 사용자가 휴일 방향을 선택한다.
4. 앱이 간단한 하루 계획을 생성한다.
5. 사용자가 계획 완료 여부를 체크한다.
6. 사용자가 만족도와 한 줄 회고를 남긴다.

초기 구현에서 반드시 필요한 것은 캘린더 연동이나 정교한 AI 추천이 아니다. 먼저 사용자가 휴일 전에 결정을 내리고, 그 결정이 실행 계획으로 이어지는 흐름을 완성한다.

## 8. 기술 방향

### 개발 언어

- HTML
- CSS
- JavaScript

### 프레임워크

- React

### 실행 환경

- 모바일 웹
- 웹 브라우저 기반 서비스

### 데이터 저장

초기 버전은 브라우저의 `localStorage`를 사용할 수 있다. Firebase, 외부 캘린더, AI API는 핵심 흐름이 검증된 뒤 확장 후보로 둔다.

### Gemini 추천 연동

Firebase Blaze 업그레이드 없이 기능을 배포하기 위해 Gemini 호출은 Vercel Function 프록시를 거친다.

Expo의 `EXPO_PUBLIC_` 환경변수는 앱 번들에 포함되므로, 클라이언트에는 프록시 URL만 넣고 Gemini API 키는 Vercel 환경변수 `GEMINI_API_KEY`로만 보관한다.

프록시 호출이 실패하면 앱은 기존 규칙 기반 추천으로 돌아간다.

선택 환경변수:

- `EXPO_PUBLIC_GEMINI_PROXY_URL`: Vercel Gemini 추천 프록시 URL
- `EXPO_PUBLIC_GEMINI_MODEL`: 사용할 Gemini 모델. 기본값은 `gemini-1.5-flash`

추천 결과에는 생성 출처 메타데이터를 함께 보관한다.

- `gemini`: Gemini 추천 성공
- `rule_based`: Gemini 실패 또는 미설정으로 기본 추천 사용
- `gemini_retry_then_rule_based`: 카테고리 변경 후 Gemini 재시도까지 실패해 기본 추천 사용

## 9. AI 구현 지침

이 문서는 기능을 확정하는 명세서가 아니라 구현 방향을 정하는 기준 문서이다.

AI가 코드를 작성할 때는 다음 순서를 따른다.

1. 먼저 "휴일 전 의사결정" 흐름이 살아 있는지 확인한다.
2. 기능을 추가하기 전에 사용자가 실제로 다음 단계로 이동할 수 있는지 확인한다.
3. 분류나 선택지는 고정 enum처럼 다루지 말고, 나중에 변경 가능한 데이터 구조로 둔다.
4. 화면을 먼저 늘리지 말고, 하나의 흐름이 끝까지 작동하도록 만든다.
5. 자동화 기능은 핵심 흐름을 완성한 뒤 추가한다.
6. 구현 중 더 나은 구조가 보이면 README의 후보 기능과 분류를 그대로 따르지 않아도 된다.

핵심 기준은 기능 개수가 아니라 사용자가 휴일 전에 결정을 끝낼 수 있는가이다.

---

## 10. 기대 효과

RestDay를 사용하면 사용자는 휴일 전에 자신의 상태를 점검하고, 막연한 휴식이 아니라 의도 있는 하루를 준비할 수 있다.

이를 통해 무기력하게 시간을 보내는 일을 줄이고, 휴식과 생산성의 균형을 맞추며, 휴일 후 만족감과 자기관리 습관을 높일 수 있다.

---

## 11. 한 줄 요약

RestDay는 휴일을 관리하는 앱이 아니라, 사용자가 휴일 전에 이번 하루를 어떻게 보낼지 결정하고 실행하게 만드는 휴일 의사결정 앱이다.
